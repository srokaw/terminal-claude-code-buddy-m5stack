# Claude Buddy — Phase 2.5: `AskUserQuestion` Mirroring — Design Spec

**Date:** 2026-05-20
**Status:** Brainstormed — ready for implementation planning
**Project:** terminal-claude-code-buddy-m5stack
**Reference:** Builds on `2026-05-18-claude-buddy-m5stack-design.md`. Phase 2 (`PermissionRequest`-based binary approval) is shipped on `phase2-permission-race` (PR #2). Mechanism + UI design were validated in a prior brainstorm (transcript `b368c529-…`); this spec captures the result for implementation.

## Goal

When Claude Code calls the `AskUserQuestion` tool (multi-choice prompts during an agent turn), the buddy device renders the questions and lets the user pick from the M5Stack's three buttons — racing the laptop the same way binary permission prompts do today. Picks on the device are returned to Claude Code as if the user typed them, with no terminal interaction required.

## Mechanism (verified)

A `PermissionRequest` hook can answer an `AskUserQuestion` by returning `decision.behavior="allow"` with `updatedInput.answers` populated:

```json
{"hookSpecificOutput": {
  "hookEventName": "PermissionRequest",
  "decision": {"behavior": "allow",
    "updatedInput": {
      "questions": [...],
      "answers": {
        "Which package manager?": "pnpm",
        "Which features?": ["A", "B"]
      }
    }}}}
```

- `answers` is keyed by **exact question text** (verbatim from `tool_input.questions[].question`).
- The value is the chosen option's `label` string (single-select) or an array of labels (multi-select).
- When the tool runs with `answers` pre-populated, Claude Code skips the native picker and returns those answers directly.

This is the same `PermissionRequest` hook plumbing Phase 2 shipped — Phase 2.5 adds an `AskUserQuestion` branch to the existing hook, rather than introducing a new hook.

Note the current state on `phase2-permission-race`: `hooks/buddy-permission-hook.py` *skips* `AskUserQuestion` (and `ExitPlanMode`) via a `SKIP_TOOLS` set, to avoid the "brief flash" bug while Phase 2 stabilized. Phase 2.5 replaces that skip with real handling.

## UI — "Direct Preset" (verified in prior brainstorm)

Three buttons map directly to options, not to a cursor. One click per answer.

### 2–3 options (the common case)

```
+----------------------------------------+
| Q 1/2  Which package manager?          |
|----------------------------------------|
|  A: npm    — Default Node.js mgr       |
|  B: pnpm   — Fast, disk-efficient      |
|  C: yarn   — Berry/classic w/ wkspaces |
+----------------------------------------+
```

- Header: `Q n/m` chip + question text (wrap to 2 lines, auto-shrink 16→14 pt if needed).
- Body: each option labeled `A:` / `B:` / `C:` with label on line 1 and one-line truncated description on line 2.
- Colors: A row tinted green-on-black, C row tinted red-on-black (preserves the binary-prompt spatial muscle memory: A on the left feels "yes-ish", C on the right feels "no-ish"). B is neutral.

### 2 options (preserve binary mapping)

Only **A** and **C** are used; B is unlabeled. The 2-option case is the most common shape after binary permissions, and the spatial habit from "A=allow / C=deny" needs to survive.

### 4 options (paged)

Page 1:
```
|  A: SvelteKit  — Lighter runtime       |
|  B: Astro      — Static-first          |
|  C: More >>                            |
```

C swaps to page 2:
```
|  A: Next.js    — SSR, App Router       |
|  B: Remix      — Nested routes         |
|  C: << Back                            |
```

This is the only hidden mode; the `More >>` row makes it discoverable.

### Multi-question (multiple `questions` in one tool call)

Each question gets its own screen. Selecting any option **auto-advances** to the next question — no explicit confirm. Top-right chip shows `Q 1/3`, `Q 2/3`, etc. After the final pick, a 1-second "Sent ✓" splash flashes (catches a wrong pick visually, no actual undo) and the device returns to the status screen.

### Multi-select (a single question with `multiSelect:true`)

Rare but real. Short-press A/B/C toggles that row's selection (the chosen rows get a `[•]` indicator). **Long-press B (≥800 ms) submits** the current selection set. When multi-select is active the screen carries a visible `B-long = submit` footer so the hidden affordance is discoverable.

### "Other" / free-text fallback

`AskUserQuestion` defines an implicit "Other" choice (free-text). With no keyboard, the device can't handle it. Affordance: **long-press A (≥800 ms)** on any ask screen = "answer on laptop / cancel device flow." The hook returns nothing (no JSON output), so Claude Code's native picker stands and the user types their `Other` answer there. Long-press A is chosen (not B) so it does not conflict with multi-select's long-press B submit.

## Bridge ⇄ device protocol additions

New bridge → device events:

```json
{"evt":"ask","id":"<uuid>","multiSelect":false,
 "questions":[
   {"text":"Which package manager?",
    "options":[{"label":"npm","desc":"Default"},
               {"label":"pnpm","desc":"Fast"},
               {"label":"yarn","desc":"Berry"}]}
 ]}
```

```json
{"cmd":"ask_cancel","id":"<uuid>"}
```

New device → bridge messages:

```json
{"cmd":"ask_answer","id":"<uuid>",
 "answers":[{"label":"pnpm"}, {"label":"A"}]}
```

`ask_answer.answers` is positional — one entry per question, in the same order as the original `questions` array. Each entry is `{"label": "..."}` for single-select or `{"labels": ["...", "..."]}` for multi-select. The hook reconstructs the `{question_text: label}` map by zipping the original questions with the response.

`ask_cancel` is sent by the hook when it times out or the user picks the long-press escape — symmetric with the existing `prompt_cancel`.

## Hook contract

`hooks/buddy-permission-hook.py` gains an `AskUserQuestion` branch:

```python
if tool == "AskUserQuestion":
    questions = tool_input.get("questions", [])
    # Send "ask_request" to bridge, await "ask_answer"
    answers_by_index = await_response(...)
    if answers_by_index is None:
        # Timeout or user pressed long-press B; yield to native prompt.
        sys.exit(0)
    answers_map = {q["question"]: a["label"] for q, a in
                   zip(questions, answers_by_index)}
    print(json.dumps({"hookSpecificOutput": {
        "hookEventName": "PermissionRequest",
        "decision": {"behavior": "allow",
                     "updatedInput": {"questions": questions,
                                      "answers": answers_map}}}}))
```

The existing binary-permission branch is untouched. `SKIP_TOOLS` shrinks to `{"ExitPlanMode"}` (Phase 2.5 handles `AskUserQuestion`; `ExitPlanMode` stays skipped — it's not a user-decision tool).

## Bridge changes

`bridge/buddy_bridge/protocol.py`:
- Add `encode_ask_request(id, questions, multi_select)`.
- Add `decode_ask_answer(line)` → `(id, [{"label":...}|{"labels":[...]}])`.
- Add `encode_ask_cancel(id)`.

`bridge/buddy_bridge/permissions.py`:
- `PermissionBroker` gains a parallel pending-ask futures dict keyed by `id`. Same lifecycle as permission futures (resolve / cancel / timeout).

`bridge/buddy_bridge/socket_server.py`:
- New `ask_request` socket message type from the hook; mirrors `permission_request` plumbing.

`bridge/buddy_bridge/__main__.py`:
- Route incoming device `ask_answer` to the broker; route `ask_cancel` from the hook to the device.

No changes to BLE link, session registry, status encoding, or any other Phase 1 plumbing.

## Firmware changes

`firmware/src/main.cpp` (or `animation.cpp` if the animation iteration lands first):

- New screen mode `ASK` parallel to existing `PROMPT` mode.
- State: `askId[40]`, `askMultiSelect`, `askQuestions[4]` (max 4 questions per call — the `AskUserQuestion` spec caps it), per-question `options[4]` with `label` + `desc`, current question index, current page, current multi-select toggles.
- Renders the Direct Preset layout above. Button handling matches the per-screen mapping (direct-select for 2–3 options, paging for 4, long-press B for escape, multi-select toggle if active).
- On final pick, send `ask_answer` and clear state. On `ask_cancel`, clear state and return to status without sending.
- Reuses the new animation primitives if the animation iteration has shipped: brief Claude `attention` while ask is on-screen, `celebrate` on submit.

## Privacy line (unchanged from base design)

- **Sent to device:** question text and option labels/descriptions — verbatim, because the user reads them to decide. (The user already sees the same text in the terminal when the native picker fires.)
- **Never sent to device:** any other tool input, transcript content, file contents, file paths unrelated to the question.

This is parity with the existing permission-prompt privacy: only what the user is being asked is sent, nothing else.

## Tests

Following the Phase 2 pattern:

- `test_permission_hook.py` — new tests: `AskUserQuestion` branch builds the correct `ask_request` payload; constructs the right `answers` map from a positional response; falls back to native (no JSON output) on timeout / cancel.
- `test_protocol.py` — new tests for `encode_ask_request`, `decode_ask_answer`, `encode_ask_cancel`.
- `test_permissions.py` — broker handles ask futures alongside permission futures without cross-contamination.
- End-to-end manual: trigger an `AskUserQuestion` from a Claude Code session, confirm device renders, A/B/C answers it, native picker stays hidden.

## Out of scope

- "Other" free-text on the device (deliberately deferred — no keyboard).
- Cursor-list or carousel UI (rejected in the prior brainstorm).
- `ExitPlanMode` mirroring (different tool semantics; out of scope).
- Animation polish (covered by the parallel animation iteration spec).

## Sequencing

This iteration is independent of the animation iteration: it touches the hook, bridge protocol, and firmware screen modes, but doesn't depend on the new sprite/morph code. If the animation iteration ships first, Phase 2.5 uses the new `react` event for a `celebrate` flash on submit; if Phase 2.5 ships first, the submit feedback is just the existing "Sent ✓" splash. Either ordering works.
